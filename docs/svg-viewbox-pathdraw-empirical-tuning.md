# SVG の viewBox と PathDraw（Preview / p-1）で有効だった調整方針

このメモは、`Preview.tsx` にインライン配置した `p-1.svg` 相当の複数 `path` を `PathDraw` で描画するときの **viewBox の決め方**について、次の経緯を整理したものです。

- **以前の状態**（Illustrator 由来の viewBox）
- **ライブラリ等で算出した幾何バウンディングボックスへ合わせた変更**（結果として不適切だったケース）
- **検証ツールで手作業調整した現状**（このデザインではこちらが最適）

---

## 状態の比較

| 観点 | 以前（初期） | 幾何 bbox ベースの変更 | 現状（手調整・理想） |
|------|----------------|-------------------------|----------------------|
| viewBox | `0 0 2448 2777.8` | `0 -561.28 2536.79 3329.66` | `0 0 2440 2950` |
| 狙い | エクスポート既定 | パス座標の理論上の外接矩形（負の Y を含む）に合わせる | レイアウト上の見え方と余白のバランス |
| 結果 | 上の余白が気になる | 意図しない表示（左上〜右下方向の線状に見える不具合など） | デザインとして最も自然 |

**現状の値**（`viewBox="0 0 2440 2950"`）は、ブラウザの開発者ツールで数値を動かしながら決めた結果として、この画面構成では最も理想的、という位置づけです。

---

## 幾何 bbox ベースのアプローチが「今回のケース」で正しくなかった理由（整理）

「パス文字列から機械的に取った min/max で viewBox を決める」という方針は、**数学的には一貫した外接矩形**を与えます。しかし次のような理由で、**見た目・レイアウト・描画パイプライン**の観点では最適解にならないことがあります。

### 1. 「余白」の正体が viewBox の外ではない

上方向の余白が、**アートボードより上に座標がはみ出しているから**とは限りません。親の高さ、`sticky` / `min-h-[100lvh]`、`content-center`、横並びの `Stick` レイアウトなど、**CSS 上の配置**で「上に空き」が生じている場合、viewBox を幾何に厳密に合わせても解消しません。

### 2. PathDraw は「パス全体の長さ × スクロール進行度」で stroke を切る

`PathDraw` は各 `path` に対して `getTotalLength()` と `stroke-dasharray` / `stroke-dashoffset` を用います（`pathDrawMath.ts`）。viewBox を変えると **画面上のスケールと縦横比**は変わりますが、**パス定義（`d` 属性）自体は不変**なので、進行度のロジックは同じです。

一方で、viewBox の縦横比や表示サイズが大きく変わると、**複数 path の見え方のバランス**や、細い線・制御点の極端に大きいベジェが占める見かけ上の領域が変わり、**「一本の斜線のように見える」**ような誤認やレンダリング上の違和感が出る余地はあります（ブラウザ・GPU・サブピクセル依存）。

### 3. 負の minY を含む viewBox は「見せる領域」が変わる

`viewBox` の **min-y を負にする**と、ユーザー座標の y &lt; 0 の部分もビューに含めます。パスが `M0,-496.3` のように負の Y から始まる場合、**クリップされていた頭頂付近が表示される**一方、**縦横比と表示ボックス内でのスケール**が変わり、周囲の余白や隣接コンテンツとの関係もまとめて変わります。  
「幾何的には正しい」が「このセクションのデザイン意図」とは一致しない、ということが起きます。

### 4. 斜めの線が出た「正確な原因」について

当該変更では、**左上から右下へ線が見えた**という報告がありました。原因をコードだけから一意に断定するには、当時のスクリーンショット・どの path が stroke 対象か・GPU/ブラウザの組み合わせなどの再現条件が必要です。

考えられる方向性としては次をメモしておきます（いずれも仮説です）。

- 極端に大きな三次ベジェ制御点を含む輪郭が、**新しいスケール・クリップ条件下で細長い弦として見える**
- **複数 path** に同一の dash 進行が乗ることで、一時的に「対角に一本」に見えるパターンになる
- **サブピクセル・アンチエイリアス**による見かけ上の線

## 有効だったアプローチ（今回の結論）

### 手元で viewBox を動かして決める（アナログ検証）

- **`0 0 2440 2950`** が、この Preview ブロックでは最も理想的、という結論になった。
- **`0 120 2440 2950`** のように **min-y を正の値にして上辺を切る**と、SVG は上にぴったり寄る一方、**下側に余白が増える**（またはバランスが崩れる）ため、このデザインでは **`0` を維持した方がよい**、というトレードオフが確認された。

つまり有効だったのは、

1. **幾何 bbox を盲信せず**、開発者ツールで `viewBox` の 4 数値を直接いじる  
2. **上下どちらの余白を許容するか**をデザイン判断する（「上を詰める」と「下が空く」はセットで検討）  
3. **`PathDraw` + 周辺レイアウト**の見え方まで含めて一度画面で確認する  

という **経験的・レイアウト統合的な調整**でした。

## 運用メモ

- 同じ SVG を **`public/images/p-1.svg` として再利用**する場合、インライン SVG の `viewBox` とファイル側の `viewBox` を揃えるかどうかは、用途（単体表示か、PathDraw ありか）で判断するとよいです。
- 「数値の正しさ（幾何）」と「画面での正しさ（知覚・レイアウト）」は別軸なので、**Preview のような複合レイアウトでは後者を優先してよい**、というのが今回の教訓です。
