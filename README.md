# Burn Your Own Style

個人のスタイルシステム（クラス、変数、スタイリングの癖）を`components`,`lib(modules)`,`scss`などの単位でpublic templateとして共有し、Claude Code・Cursor 等エージェント向けドキュメントを充実させることで、Web 制作の全工程をAI前提で進める為のプロジェクトです。

**現在は`react`ブランチで進行中。完成後にmainにマージ予定**

## Unitとは

Unitは、CSSクラス名・スキル名・コンポーネント名を兼ねるシステム固有の概念です。

## 思想

- LLM の学習データに基づくwebデザイン・コーディングは平均的で、振れ幅の大きい、標準ではないものであり、個人のマークアップ、スタイリングとかけ離れたものになる。
- 事務作業のように決められた手順を実行させることで、vibeコーディングツールでは創造できないプロダクトを効率的に実装できる。
- 2026年3月時点のトレンド「AIの自立性強化」とは正反対のアプローチであるが、ウェブ制作のコモディティー化は業界全体の緩やかな品質低下につながると考え、もう一つのトレンド「ハーネスエンジニアリング」に近い一つのアプローチとして、ニッチでハイクオリティな需要を満たすための環境構築に努める。

## 目的

- 個人のクラス、変数、スタイリングの癖を再利用可能にする
- 「AI臭い」デザインをさせないようマークアップレベルで制御する
- エージェントによるデザイン再現の精度と効率を向上させる
- 一貫性のあるUI/UXを維持しながら開発スピードを上げる

## 製作者の開発環境

個人の理想の環境構築には多くの時間を費やす必要を感じています。チーム内で共有する場合もメンバーが暗黙知を理解するというコストが発生します。ここでは私の偏ったスタイルシステムでの例をご紹介します。

### 設計思想

- ユーザーのハンドコーディングと明示的スキル実行に備え速記性を重視する。
- UnitではパスカルケースでTailwindと視覚的に差別化
- デスクトップでの外観にフォーカスしたクラス名を使用。ブレイクポイントをTailwindに合わせて上書きも可能

### プロジェクト構成


### Unit一覧

| Unit | 説明 |
|------|------|
| `Cards` | 一般的な横並びコンテナ（2〜5列程度） |
| `Panel` | 一般的な縦並びコンテナ。ImgTextを複数まとめたものに近い |
| `ImgText` | 画像+テキスト横並びコンテナ。画像比率をValueクラスで可変 |
| `Toggle` | summary,detailsタグを使う開閉コンテンツ。特殊なCSSが必要なためUnit化 |
| `FlexR` | クラス名が比率を表す。2種類のコンテンツの比率を制御するラッパー |

### Valueクラス

CSS変数に値を渡すクラス。

| クラス | 意味 | 使用Unit |
|--------|------|----------|
| `.col2`, `.col3`, `.col4` | カラム数 | Cards |
| `.img20`, `.img30`, `.img40`, `.img60` | 画像比率 | ImgText, Panel |

### Modifierクラス

Unitクラス毎に定義されている状態・モードの切り替え（パスカルケース）。

| クラス | 効果 | 使用Unit |
|--------|------|----------|
| `.IsRev` | 画像の位置を反転（右に） | ImgText, Panel |
| `.IsLayer` | 画像とテキストを重ねる | Cards |
| `.IsFlow` | パネル間に矢印 | Panel |
| `.IsQa` | Q&Aの装飾付与 | Toggle |
| `.IsGrow` | スマホで自動調整（flex:1） | Cards |
| `.IsFix` | 固定幅 --itemW指定 | Cards |

### FlexRのValueクラス

クラス名が比率を表す。2種類のコンテンツの比率を制御するラッパー。

| クラス | 左：右比率 | 説明 |
|--------|-----------|------|
| `.Flex55` | 50%:50% | 等分 |
| `.Flex46` | 40%:60% | 左4、右6 |
| `.Flex64` | 60%:40% | 左6、右4 |
| `.Flex37` | 30%:70% | 左3、右7 |
| `.Flex73` | 70%:30% | 左7、右3 |
| `.Flex28` | 20%:80% | 左2、右8 |
| `.Flex82` | 80%:20% | 左8、右2 |

25%など細かい調整は近いクラスを選び`--few`を上書きする。

### 使用例

**Cards（3カラム）**:
```html
<div class="Cards col3">
  <div class="item gap-0 ">
    <figure><img src="/images/960x480.png" alt=""></figure>
    <div class="p-4 rounded-lg shadow-lg flex-1">
      <div>
        <h3 class="font-bold mb-2">カード 1</h3>
        <p>ここで Tailwind クラスを使って装飾できます。</p>
      </div>
    </div>
  </div>
  ...
</div>
```

**Toggle（アコーディオン）**:
```html
<details class="Toggle IsQa">
  <summary class="">これはどのような仕組みですか？</summary>
  <div class="">
    <p>summary と details タグを使った開閉コンテンツです。</p>
  </div>
</details>
```

Tailwindのarbitrary value構文を使用：

- `bg-[var(--MC)]` - メインカラーの背景
- `text-[var(--TC)]` - テキストカラー
- `border-[var(--SC)]` - サブカラーのボーダー

## 変数設計

### 基本レイアウト変数

| 変数 | 例 | 説明 | 使用シーン |
|------|----------|------|-----------|
| `--wid` | `1080px` | `main>*`のwidthになる幅 | コンテンツ幅制御 |
| `--gap` | `2rem` | Flexコンテンツの基本ギャップ | Cards, Panel, FlexR系 |
| `--MY` | `4rem` | セクション間の基本縦余白 | `main>*+*`, `.wrapper>*+*` |
| `--PX` | `1.5rem` | 全コンテンツの基本横余白 | `main`のpadding-inline |
| `--into` | 計算値 | 中身が--widの幅になる横余白 | `.intoL`, `.intoR` |
| `--out` | 計算値 | コンテンツ幅を画面幅にする | `.out`, `.outR`, `.outL` |
| `--imgW` | `30%` | 画像幅比率 | ImgText, Panel |
| `--itemW` | 計算値 | Cardsアイテム幅 | `.Cards`クラス |
| `--few` | `30%` | FlexRの小さい要素幅（各Flexクラスで既定値が入り、上書き可） | Flex46, Flex37など |

## システム開発、実践に基づく考察
- `pencil.dev` のアーキテクチャが優れていると感じる。思ったより簡単な意思伝達が可能だが、デザイン作成自体が特殊な作業であり**注意事項**が膨れ上がる。段階的開示で対応してもコンテキストは多く消費する。
- 「構造化データ」を作るという意味で本番用コンポーネントのプロトタイピングをすることと変わらない。pencilやstitchは「デザインまでの作業」「デザイン段階の共有」「AI開発自体のエンタメ化」という視点で優れている。
